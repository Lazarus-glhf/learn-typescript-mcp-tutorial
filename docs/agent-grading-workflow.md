# AI Agent 打分流程

本教程要求学习者每完成一章后，把练习提交给 AI Agent 进行打分。Agent 不只看代码片段，而是根据项目文件、命令输出、练习说明和自检结果综合评分。

## 学习者提交内容

每次提交必须包含：

1. 当前章节编号，例如 `chapter-00`
2. 项目目录结构
3. 本章修改过的文件内容或 Git diff
4. 实际运行过的命令
5. 命令输出
6. 自检清单
7. 遇到的问题或不确定点

推荐使用模板：

```text
templates/agent-submission.md
```

## Agent 评分原则

Agent 必须基于证据评分，不要凭感觉给满分。

### 可以作为证据的内容

- 文件树
- 源码内容
- `package.json`
- `tsconfig.json`
- 终端命令输出
- 测试输出
- 编译输出
- 学习者对错误的说明

### 不足以作为证据的内容

- “我已经完成了”
- 没有命令输出的截图式描述
- 缺少文件内容的口头总结
- 明显没有实际运行的推测结果

## 标准评分结构

Agent 回复必须包含：

```text
总分：xx / 100
等级：通过 / 需要修改 / 建议重做

评分明细：
- 正确性：xx / 40
- 类型使用：xx / 20
- 项目结构：xx / 15
- 可运行性：xx / 15
- 代码清晰度：xx / 10

主要问题：
1. ...
2. ...

修改建议：
1. ...
2. ...

下一步：
- 可以进入下一章 / 修改后重新提交
```

## 等级定义

| 分数 | 等级 | 含义 |
|---|---|---|
| 90-100 | 通过，优秀 | 可以进入下一章，只有小建议 |
| 75-89 | 通过 | 可以进入下一章，但建议先修正反馈 |
| 60-74 | 需要修改 | 不建议进入下一章，应按反馈修改后重交 |
| 0-59 | 建议重做 | 核心目标未达成，需要重新完成本章 |

## Agent 反馈风格

Agent 应该：

- 直接指出问题，不含糊
- 用文件名、代码片段或命令输出定位问题
- 给出可执行的修改建议
- 不替学习者直接写完整答案，除非学习者明确请求讲解
- 对新手解释为什么错，而不是只说错了

Agent 不应该：

- 因为学习者是初学者就降低核心正确性标准
- 没有证据就判断可运行
- 把下一章知识强塞进当前章评分
- 对风格问题扣过多分，除非影响理解或维护

## 章节 Rubric

每章可以有独立 rubric，放在：

```text
docs/rubrics/chapter-XX-rubric.md
```

如果章节 rubric 与本文件冲突，以章节 rubric 为准。
